# SPIKE: Bun --compile Static Asset Embedding with @elysiajs/static

**Bead**: jobapptracker-gel
**Date**: 2026-03-27
**Timebox**: 2 hours
**Bun version tested**: 1.3.11
**Unblocks**: jobapptracker-a92 (Package standalone binary for distribution)

## Summary

`@elysiajs/static` is **NOT compatible** with `bun build --compile` binaries. A custom `serveEmbedded()` helper using `import ... with { type: "file" }` is the recommended replacement for compiled binaries.

## Findings

### 1. @elysiajs/static is CWD-dependent

The staticPlugin resolves asset directories via `path.resolve(assets)`, which is relative to `process.cwd()`. In production mode (`alwaysStatic: true`), it scans the directory at startup using `listFiles(path.resolve(assets))`. In development mode, it does `fs.stat()` at request time.

Neither mode reads from Bun's embedded virtual filesystem (`$bunfs`). When a compiled binary runs from a directory without the expected `public/` (or `src/presentation/styles/`, etc.), staticPlugin returns 404 for all assets.

**Verified**: Compiled binary serving via staticPlugin returns ENOENT + 404 when run from any directory that lacks the original asset folders on disk.

### 2. Bun's `import ... with { type: "file" }` works reliably

Files imported with the `{ type: "file" }` attribute are embedded into the binary and accessible at runtime via `Bun.file(importedPath)`. At runtime, the import resolves to a `$bunfs/root/...` virtual path.

```typescript
import cssFile from "./public/style.css" with { type: "file" };
// At runtime in compiled binary: cssFile === "/$bunfs/root/style-xcbbwxec.css"
// Bun.file(cssFile).text() returns the CSS content
```

**Verified**: Works from any CWD. The file is fully embedded in the binary.

### 3. `--asset-naming` controls embedded file paths

| Flag | Import path at runtime | `Bun.embeddedFiles` name |
|------|----------------------|--------------------------|
| (default) | `/$bunfs/root/style-xcbbwxec.css` | `style-xcbbwxec.css` |
| `--asset-naming "[dir]/[name].[ext]"` | `/$bunfs/root/public/style.css` | `public/style.css` |

Use `--asset-naming "[dir]/[name].[ext]"` to preserve directory structure and original filenames.

### 4. `Bun.embeddedFiles` is unreliable with large bundles

When bundled with elysia (~315 modules), `Bun.embeddedFiles` returns an empty iterator despite the files being successfully embedded and accessible via their import paths. With a minimal bundle (2-3 modules), `Bun.embeddedFiles` works correctly.

**Conclusion**: Do NOT rely on `Bun.embeddedFiles` for asset discovery in production. Use explicit import paths instead.

### 5. Binary size

A minimal Elysia server with 2 embedded CSS files produces a ~96MB binary. This is the Bun runtime overhead and is independent of asset count.

## Recommended Strategy for jobAppTracker

### Build command

```bash
bun build --compile src/index.ts --outfile dist/jobapptracker
```

No special asset flags are needed -- `import ... with { type: "file" }` handles embedding automatically.

### Asset registration module

Create a build-time-generated asset manifest that imports all static files:

```typescript
// src/presentation/embedded-assets.ts (generated by build script)
import styles_main from "../presentation/styles/main.css" with { type: "file" };
import styles_dashboard from "../presentation/styles/dashboard.css" with { type: "file" };
import scripts_featureFlags from "../presentation/scripts/feature-flags.js" with { type: "file" };
import scripts_pipelineClient from "../presentation/scripts/pipeline-client.js" with { type: "file" };
// ... all other static files

export const embeddedAssets: Record<string, string> = {
  "styles/main.css": styles_main,
  "styles/dashboard.css": styles_dashboard,
  "scripts/feature-flags.js": scripts_featureFlags,
  "scripts/pipeline-client.js": scripts_pipelineClient,
};
```

### Serve helper (replaces staticPlugin for compiled mode)

```typescript
// src/application/server/serveEmbedded.ts
import { Elysia } from "elysia";

const MIME_TYPES: Record<string, string> = {
  ".css": "text/css",
  ".js": "application/javascript",
  ".png": "image/png",
  ".svg": "image/svg+xml",
  ".json": "application/json",
  ".html": "text/html",
  ".ico": "image/x-icon",
  ".woff2": "font/woff2",
};

function getMime(path: string): string {
  const ext = path.slice(path.lastIndexOf("."));
  return MIME_TYPES[ext] ?? "application/octet-stream";
}

export function serveEmbedded(
  prefix: string,
  assets: Record<string, string>,
): Elysia {
  return new Elysia().get(`${prefix}/*`, ({ params }) => {
    const filePath = assets[params["*"]];
    if (!filePath) return new Response("Not Found", { status: 404 });
    return new Response(Bun.file(filePath), {
      headers: {
        "Content-Type": getMime(params["*"]),
        "Cache-Control": "public, max-age=86400",
      },
    });
  });
}
```

### Server integration (dual-mode)

```typescript
// In startElysiaServer.ts
import { embeddedAssets } from "../presentation/embedded-assets";

const isCompiled = process.argv[0]?.includes("$bunfs") ||
  !import.meta.dir.startsWith("/");

if (isCompiled) {
  app
    .use(serveEmbedded("/styles", filterByPrefix(embeddedAssets, "styles/")))
    .use(serveEmbedded("/scripts", filterByPrefix(embeddedAssets, "scripts/")))
    .use(serveEmbedded("/assets", filterByPrefix(embeddedAssets, "assets/")));
} else {
  app
    .use(staticPlugin({ assets: "src/presentation/styles", prefix: "/styles" }))
    .use(staticPlugin({ assets: "src/presentation/scripts", prefix: "/scripts" }))
    .use(staticPlugin({ assets: "src/presentation/assets", prefix: "/assets" }));
}
```

### Build script changes

The `build` script in `package.json` needs to:
1. Run `build:scripts` (existing step -- compiles TS to JS for browser)
2. Generate `embedded-assets.ts` by scanning the three asset directories
3. Run `bun build --compile src/index.ts --outfile dist/jobapptracker`

A generator script can scan `src/presentation/{styles,scripts,assets}` and produce the import manifest automatically.

## What works / What doesn't

| Approach | Compiled binary | Dev mode | Verdict |
|----------|:-:|:-:|---------|
| `@elysiajs/static` staticPlugin | No (CWD-dependent) | Yes | Dev only |
| `import ... with { type: "file" }` + `Bun.file()` | Yes | Yes | Recommended for compile |
| `Bun.embeddedFiles` iterator | Unreliable (empty with large bundles) | N/A | Avoid |
| HTML imports (`import page from "./index.html"`) | Yes (auto-bundles frontend) | Yes | Alternative for HTML-first apps |

## Gotchas

1. **Each file needs an explicit import statement** -- there is no wildcard/glob import syntax. A build-time codegen step is needed to enumerate assets.
2. **`Bun.embeddedFiles` is unreliable** with large dependency trees (confirmed empty with elysia's 315-module bundle on Bun 1.3.11). May be fixed in future Bun versions.
3. **Content hashes in filenames** are the default. Use `--asset-naming "[dir]/[name].[ext]"` if stable filenames are needed, or map imports to URL paths explicitly.
4. **Binary size** is ~96MB minimum (Bun runtime). Asset files add negligible overhead.
5. **The `$bunfs` virtual filesystem** is read-only. `fs.existsSync()` and `fs.statSync()` work on `$bunfs` paths, but directory listing (`readdirSync`) does not.

## Open Questions for Implementation

1. **Compiled-mode detection**: What is the most reliable way to detect if running as a compiled binary? Options: check `process.argv[0]`, check `import.meta.main` behavior, or use an env var set by the build script.
2. **Asset manifest generation**: Should this be a Bun macro, a pre-build script, or integrated into the existing `build:scripts` step?
3. **Cache headers and ETags**: The current staticPlugin handles ETags automatically. The embedded approach needs explicit ETag generation (use content hash from `--asset-naming` or compute at startup).
